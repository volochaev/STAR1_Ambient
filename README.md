## 2. Архитектура системы освещения

### 2.1. Зоны

Каждый блок содержит до 4 зон:

| Zone ID | Описание |
|---------|----------|
| `WS_ZONE_STRIP` | Основная длинная линия |
| `WS_ZONE_HANDLE` | Подсветка ручки |
| `WS_ZONE_STORAGE` | Ниша/отделение |
| `WS_ZONE_FOOTWELL` | Подсветка ног |

Каждая зона имеет:
- Свою длину LED
- Отдельный цвет/палитру
- Своё поведение в intro/outro
- Свой brightness scale

### 2.2. Система палитр

Палитры состоят из цветовых стопов (color stops):

```c
typedef struct {
    float pos;      // 0.0 .. 1.0
    uint8_t r, g, b;
} ws_pal_stop_t;
```

Палитры используются в темах и эффектах для создания градиентов.

**Доступные палитры:**
- Sunset Amber, Ocean Blue, Red Moon, Purple Silk
- Glacier, Energize, Polar White, Night Blue
- Hyacinth, Copper Gold, Rose

### 2.3. Эффекты (FX Engine)

**Однократные эффекты (oneshot):**
- `FX_WELCOME` / `FX_MB_WELCOME` - приветственная анимация
- `FX_GOODBYE` / `FX_MB_GOODBYE` - анимация выключения

**Постоянные эффекты (scene effects):**
- `FX_GRADIENT_FLOW` - плавный градиент
- `FX_TWIN_WAVE` - двойная волна
- `FX_MB_OCEAN_FLOW` - океанский поток
- `FX_MB_ENERGIZE_PULSE` - энергетический пульс
- `FX_DUAL_ZONE` - двойная зона
- И другие...

### 2.4. Scene Player (машина состояний)

**Состояния:**

```
PST_IDLE → PST_INTRO → PST_BRIDGE → PST_SCENE → PST_OUTRO → PST_IDLE
```

- **PST_IDLE** - бездействие
- **PST_INTRO** - вступительная анимация (oneshot)
- **PST_BRIDGE** - переходный кадр
- **PST_SCENE** - активная сцена (непрерывная)
- **PST_OUTRO** - завершающая анимация (oneshot → idle)

**Функции Player:**
- Управление яркостью (`calc_brightness`)
- Night mode
- Плавные переходы всех зон
- Смена тем

## 3. Драйвер WS2812B

### Технические характеристики

- **Метод передачи:** TIM1 PWM (4 канала) + DMA
- **Частота:** 800 kHz
- **Формат данных:** GRB (Green, Red, Blue)
- **Количество зон:** до 4 независимых каналов

### Использование

1. Инициализация: `ws_init()` с указанием буферов
2. Установка пикселей: `ws_set_pixel_rgb()`
3. Отправка кадра: `ws_render()`
4. Обработка DMA: `ws_dma_tc_isr()` в `HAL_TIM_PWM_PulseFinishedCallback()`

**Примечание:** DMA-буферы должны иметь размер: `(led_count * 3 * 8) + WS_RESET_SLOTS` слов

## 4. CAN-система

### 4.1. Общая архитектура

Система состоит из нескольких блоков (FL, FR, RL, RR, Dashboard, Rear), подключенных к одной CAN-шине автомобиля.

**Роли:**
- **MASTER** - один блок (автоматически определяется), читает CAN от машины, управляет системой
- **SLAVE** - остальные блоки, принимают команды от master и синхронизируют состояние

### 4.2. Определение мастера

✅ **Реализовано:** Все блоки используют одну прошивку. Master определяется автоматически.

**Алгоритм:**

1. **Discovery механизм:**
   - Каждый блок отправляет discovery-пакет каждые 1000 мс
   - Пакет содержит: BOARD_TYPE, unique_id, текущая роль

2. **Приоритеты плат:**
   ```
   FL > FR > DASHBOARD > RL > RR > REAR
   ```
   Master выбирается автоматически по приоритету.

3. **Heartbeat механизм:**
   - Master отправляет sync-пакет каждые 250 мс
   - Master отправляет extended-пакет каждые 250 мс (при extended режиме)
   - Если master не отвечает > 1 сек → автоматический failover

4. **Отказоустойчивость:**
   - При отказе master, slave с наивысшим приоритетом становится новым master
   - При восстановлении, роль автоматически возвращается к платформе с более высоким приоритетом
   - Система продолжает работать даже при отказе одной из плат

### 4.3. CAN ID

Используется только 2 CAN ID:

| ID | Описание | Отправитель |
|----|----------|-------------|
| `0x351` | OEM ambient от IC (яркость, цвет) | Машина → Все |
| `0x353` | Unified Master Protocol | Master/Slaves |

**Типы пакетов 0x353 (различаются старшими 4 битами Byte0):**

| Тип пакета | Byte0 (старшие 4 бита) | Описание |
|------------|------------------------|----------|
| `PKT_TYPE_DISCOVERY` | `0x00` | Discovery пакет (от всех плат) |
| `PKT_TYPE_MASTER` | `0x10` | Master пакет (полное состояние) |
| `PKT_TYPE_SYNC` | `0x20` | Sync/Heartbeat пакет |
| `PKT_TYPE_EXT` | `0x30` | Extended синхронизация |

### 4.4. Форматы пакетов

#### Discovery Packet (PKT_TYPE_DISCOVERY)

**ID:** `0x353`, **DLC:** 8

| Byte | Описание |
|------|----------|
| 0 | `0x00 \| BOARD_TYPE` (0-5: FL,FR,RL,RR,DASHBOARD,REAR) |
| 1-4 | `unique_id` (уникальный идентификатор платы) |
| 5 | Текущая роль (0=slave, 1=master) |
| 6-7 | Reserved |

**Периодичность:** Все платы каждые 1000 мс

---

#### Master Packet (PKT_TYPE_MASTER)

**ID:** `0x353`, **DLC:** 8

| Byte | Описание |
|------|----------|
| 0 | `0x10 \| flags` (bit0=EXTENDED_MODE, bit1=NIGHT_MODE) |
| 1 | `bank_id` (0=auto, 1=amber, 2=blue, 3=white) |
| 2 | `theme_index` (индекс темы в банке) |
| 3 | `oem_color` (0=Amber, 1=Blue, 2=White) |
| 4 | `brightness_raw` (0..5) |
| 5-7 | Reserved |

**Периодичность:** Master каждые 100 мс при изменении состояния

---

#### Sync Packet (PKT_TYPE_SYNC)

**ID:** `0x353`, **DLC:** 8

| Byte | Описание |
|------|----------|
| 0 | `0x20 \| bank_id` |
| 1 | `theme_index` |
| 2-7 | Reserved |

**Периодичность:** Master каждые 250 мс

**Назначение:** Heartbeat для отслеживания работоспособности master

---

#### Extended Packet (PKT_TYPE_EXT)

**ID:** `0x353`, **DLC:** 8

| Byte | Описание |
|------|----------|
| 0 | `0x30 \| flags` (bit0=EXTENDED_MODE, bit1=NIGHT_MODE) |
| 1 | `bank_id` |
| 2 | `theme_index` |
| 3-7 | Reserved |

**Периодичность:** Master каждые 250 мс (только при extended режиме)

---

#### OEM Packet

**ID:** `0x351`, **DLC:** 8

| Byte | Описание |
|------|----------|
| 0 | `brightness_raw` (0..5) |
| 1 | Reserved |
| 2 | `oem_color` (0=Amber, 1=Blue, 2=White) |
| 3-7 | Reserved |

**Отправитель:** Машина (IC) → Все платы  
**Обработка:** Только master платой

**Переключение Extended режима:**
- Смена цвета **5 раз** через CAN 0x351 в течение **3 секунд**
- Переключение происходит только на master плате
- При выходе из Extended режима сбрасываются `bank_id` и `theme_index`

### 4.5. Сохранение настроек в Flash

**Расположение:** Последняя страница Flash (адрес `0x0801F800`, 2KB)

**Сохраняемые настройки:**
- `extended_mode` - режим работы (OEM/Extended)
- `bank_id` - выбранный банк тем
- `theme_index` - индекс темы в банке

**Механизм:**
1. Загрузка при инициализации (`can_ambient_init()`)
2. Сохранение через 2 секунды после последнего изменения
3. Сохранение только на master плате
4. Защита: Magic number (`0x53544152` = "STAR") + CRC32

**Примечания:**
- Flash имеет ограниченное количество циклов записи (~10,000)
- При повреждении данных используются значения по умолчанию

## 5. Поток данных

### Инициализация

1. Все платы стартуют и отправляют discovery-пакеты
2. Плата с наивысшим приоритетом (FL) становится master
3. Master начинает отправлять sync-пакеты каждые 250 мс

### Нормальная работа

**Master:**
- Читает CAN от машины (0x351)
- Вычисляет bank, theme_index, brightness
- Отправляет master/sync/ext/discovery пакеты (0x353)
- Сохраняет настройки в Flash

**Slave:**
- Читает master/sync/ext пакеты от master
- Применяет bank/theme/brightness из пакетов
- Отправляет discovery-пакеты каждые 1000 мс

**Lighting engine:**
- Выполняет intro/scene/outro на всех платах синхронно

### Failover (отказоустойчивость)

1. Master перестает отвечать (нет sync/extended пакетов > 1 сек)
2. Slave с наивысшим приоритетом обнаруживает timeout
3. Slave автоматически переходит в master режим
4. Новый master начинает отправлять пакеты
5. При восстановлении старого master, роль возвращается автоматически

## 6. Работа системы

### OEM режим (машина управляет)

1. Автомобиль отправляет цвет/яркость (0x351)
2. Master читает 0x351 и выбирает дефолтную тему для банка
3. Master отправляет 0x353 (PKT_TYPE_MASTER) каждые 100 мс
4. Master отправляет 0x353 (PKT_TYPE_SYNC) каждые 250 мс
5. Все блоки синхронизируются и применяют одинаковую тему

### Extended режим

1. Master вычисляет bank/theme_index (локально или из внешних команд)
2. Master отправляет 0x353 (PKT_TYPE_MASTER) каждые 100 мс
3. Master отправляет 0x353 (PKT_TYPE_EXT) каждые 250 мс
4. Все блоки применяют ту же тему синхронно

## 8. Настройка плат

### Определение BOARD_TYPE

В `main.c` включите нужный файл **ПЕРЕД** включением `ambient.h`:

```c
#include "board_door_fl.h"      // Front-Left door
// или
#include "board_door_fr.h"      // Front-Right door
// или
#include "board_door_rl.h"      // Rear-Left door
// или
#include "board_door_rr.h"      // Rear-Right door
// или
#include "board_dashboard.h"    // Dashboard
// или
#include "board_rear.h"         // Rear ambient
```

Каждый файл автоматически определяет `BOARD_TYPE`.

### Принудительная установка роли (опционально)

```c
#define AMB_ROLE_MASTER  1  // Принудительный master
#define AMB_ROLE_MASTER  0  // Принудительный slave
// Не определять = автоматическое определение (по умолчанию)
```

## 9. Инструкции по прошивке

### Шаг 1: Выбор типа платы

Убедитесь, что в `main.c` включен правильный `board_xxx.h` файл (см. раздел 8).

### Шаг 2: Настройка демо режима (опционально)

По умолчанию демо режим отключен. Для включения:

```c
#define DEMO_MODE  1  // в main.c
```

**Примечание:** Демо режим работает только на master плате и предназначен для тестирования.

### Шаг 3: Компиляция и прошивка

1. Откройте проект в STM32CubeIDE
2. Проверьте правильность `board_xxx.h` в `main.c`
3. Скомпилируйте проект (Build → Build Project)
4. Подключите плату через ST-Link
5. Прошейте плату (Run → Debug или Run → Run)

### Шаг 4: Проверка после прошивки

После прошивки плата должна:
1. Автоматически определить свою роль (master/slave)
2. Начать отправлять discovery-пакеты каждые 1000 мс
3. Если master: читать CAN (0x351), отправлять master/sync/ext пакеты
4. Если slave: слушать пакеты от master и синхронизировать состояние
